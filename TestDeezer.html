L'idal serait d'ouvrir l'intgralit des fichiers et de crer une norme matrice reprsentant le profil de chaque utilisateur au cours du temps. Le problme c'est que cela devient vite ingrable au niveau de la mmoire vu la quantit de donnes rcupres par jour. Il faut une autre approche.

On peut crer une matrice journalire pour chaque fichier. En lignes, les utilisateurs, en colonnes, les tracks et les artistes. L'historique des coutes au cours du temps est distribue dans plusieurs fichiers tout en conservant la possibilit de dtecter des changements dans les habitudes d'coute des diffrents utilisateurs.

Le calcul peut se faire de manire distribue en utilisant les algorithmes de type Map-Reduce propos par des plateformes comme Hadoop ou Spark. Les calculs peuvent trs bien tre parallliss (par exemple via des containers Docker) ce qui permet de rendre cette mthode viable en chane de production. Voir le schma ci dessous.

Il faut alors stocker un profil total des utilisateurs dans une matrice ou dans des vecteurs distribus pour chaque utilisateurs. Pour le mettre jour il suffit de calculer la matrice journalire et de sommer la ligne cre.

Création de la matrice journalière¶

La matrice journalire est cre partir de toutes les coutes stockes dans le fichier concern. On regroupe les lignes par utilisateur ce qui nous permet de retrouver tous les tracks et/ou les artistes qu'il a cout. On effectue ensuite un changement de reprsentation des tracks et des artistes en passant d'un reprsentation en chaine de charactres une reprsentation numrique Bag of Words. Cela permet de compter simplement le nombre d'occurences d'coute d'un track.

Ensuite, pour aller plus loin on ralise une normalisation TF-IDF (term frequency inverse document frequency) qui permet de mettre en valeur les tracks rares ou contrario de donner un poids plus faible aux tracks populaires. En effet, les recommandations les plus pertinentes sont celles qui concernent les tracks rares plutt que les tracks populaires.

def create_df_normalized_on_day(day_num, ptracks, partists, path_to_file="./logs/"):
    """
    Create matrix from file for given day
    @return 
        - matrix correspond to artists and tracks normalized for all users
        - list of columns name correspond to artist and tracks ids.
        - users informations and their ids correspond to index in matrix
    """
    with open(path_to_file+"logs_day_{}".format(day_num), 'r') as f:
        list_of_logs_day1 = f.read().splitlines() 
        

    df_day1 = pd.DataFrame([{"user_id":elt.split("|")[0], "country":elt.split("|")[1], "artist_id":elt.split("|")[2], "track_id":elt.split("|")[3]} for elt in list_of_logs_day1])
    df_grouped = df_day1.groupby("user_id")
    df_day_agg_track = df_grouped["track_id"].agg(lambda x: ' '.join(x))
    df_day_agg_artist = df_grouped["artist_id"].agg(lambda x: ' '.join(x))
    vectorizer_tracks = CountVectorizer()
    vectorizer_tracks.fit(ptracks)
    tfidf_transformer_tracks = TfidfTransformer()
    vectorizer_artists = CountVectorizer()
    vectorizer_artists.fit(partists)
    tfidf_transformer_artists = TfidfTransformer()
    
    X_tracks = tfidf_transformer_tracks.fit_transform(vectorizer_tracks.transform(df_day_agg_track))
    X_artists = tfidf_transformer_artists.fit_transform(vectorizer_artists.transform(df_day_agg_artist))
    columns = ['track_'+elt for elt in  vectorizer_tracks.get_feature_names()]+ ['artist_'+elt for elt in vectorizer_artists.get_feature_names()]
    return hstack((X_tracks, X_artists)).tocsr(), columns,df_day1.groupby("user_id", as_index=False).first().drop(["artist_id","track_id"], axis=1)

Pour optimiser l'empreinte mmoire de ces donnes on utilise des matrices sparses (creuses). Cela permet de ne garder que les positions et les valeurs correspondantes ces dernires. La taille mmoire est alors trs rduite ce qui est trs important dans des problme d'analyse "smantique". En effet, la taille du vocabulaire (ici le nombre de musique dans la base de Deezer) est trs important par rapport aux coutes moyennes d'un utilisateur.

La matrice sparse COO est une matrice codant les coordonnes de chaque point ainsi que la valeur correspondante alors qu'une matrice CSR est une matrice condense par ligne. Ce type de matrice est avantageux pour faire de l'indexation par ligne (alors qu'elle est beaucoup plus lente par colonne). C'est ce type de matrice qui est retenu pour l'tude.

def get_list_sum(user_id,all_t, all_a, first=1, last=10):
    """
    get the sum of one user from his id between two dates
    """
    list_sum = []
    for i in range(first,last):
        list_sum.append(create_df_normalized_on_day(i, all_t, all_a)[0].tocsr()[user_id].toarray().sum())
    return list_sum

def get_profil_from_day_to_day(user_id,all_t, all_a,first=1, last=10):
    """
    get the profil of user between two dates
    """
    final_profil = None
    for i in range(first,last):
        if(final_profil!=None):
            res = create_df_normalized_on_day(i, all_t, all_a)[0][user_id]
            final_profil = final_profil + res
        else:
            final_profil = create_df_normalized_on_day(i, all_t, all_a)[0][user_id]
    return final_profil

Nous souhaitons proposer aux utilisateurs de pouvoir follow sur le site des utilisateurs ayant des gots musicaux similaires. Nous supposons que nous disposons dune matrice M tel que M(i,j) est une mesure de similarit entre lutilisateur i et lutilisateur j.

a. Dfinir une mesure de similarit. Justifiez.

Il peut exister plusieurs mtriques pour calculer la similarit entre deux utilisateurs.

  • Une pourrait tre la distance entre les deux vecteurs de profil de la matrice cre dans l'exercice 1, c'est une simple norme. On value la distance entre chaque tracks pour les deux utilisateurs. Plus les deux utilisateurs ont les mmes habitudes d'coutes plus la norme sera faible. On pourra alors retrouver les utilisateurs ayant la plus faible distance donc tant les plus similaires.
  • Une deuxime pourrait tre la corrlation entre deux variables statistiques modlises par les deux vecteurs crs par les coutes des deux utilisateurs.
  • Une troisime pour tre une similarit se basant sur la thorie des recommandations collaboratives. C'est celle que l'on va dtailler.

Ici les poids sont reprsents par coutes normalises des diffrents utilisateurs plus que par une note attribu ou une valeur binaire. Deezer utilise deux manires de rcuprer ces informations, de faon passive et de faon active du point de vu de l'utilisateur. La manire passive est celle expliqu dans l'exercice 1, cela reprsente les coutes des diffrents utilisateurs. La deuxime (active) est celle qui passe par leurs likes et/ou leurs playlists. Le problme ici est que la plupart des donnes sont manquantes puisques les utilisateurs ne notent ou n'coutent que trs peu de tracks sur la quantit totale propose par Deezer. Pour palier ce problme on utilise une mthode des plus proche voisin qui utilise une mtrique de similarit n'utilisant que les valeurs en communs.
Il existe alors plusieurs mtrique de distance pouvant tre utilises :

  • Jaccard : taille de l'intersection sur la taille de l'union. Cette mesure de similarit n'tant qu'une mesure de cardinalit ne permet pas de prendre en compte la normalisation TF-IDF. https://docs.scipy.org/doc/scipy/reference/generated/scipy.spatial.distance.jaccard.html
  • Cosine : cette mesure prend en compte le produit scalaire et la norme des deux vecteurs. http://scikit-learn.org/stable/modules/generated/sklearn.metrics.pairwise.cosine_similarity.html

La mesure de similarit utilisant le cosinus est plus pertinente dans cette dmarche puisqu'elle permet de prendre en compte la normalisation (term frequency inverse document frequency). En effet, cette mesure est trs utilis dans l'analyse de textes. La mesure de Jaccard peut tre utilis si l'on ne normalise pas et que l'on ne note que pas 0,1 si l'utilisateur cout/aim la chanson ou pas.

La taille maximale de cette matrice est de N avec N = ~16M, le nombre d'utilisateurs. C'est une matrice symtrique puisque M(i,j) = M(j,i). Cette relation vient de la dfinition mme d'une distance qui doit tre symtrique (soit d(x,y) = d(y,x)). Cette matrice est aussi dfinie positive. La proprit de parcimonie dpend de la mesure de similarit. Mais je ne peux pas a priori faire d'hypothse de parcimonie. Pour que la matrice soit pleine il faudrait uniquement que chaque utilisateur ait au moins cout un tracks en commun avec un autre utilisateur.
D'aprs la proprit de symtrie il n'est pas ncessaire de stocker l'intgralit de la matrice. Il suffit de stocker la matrice basse sans la diagonale cela permet de diviser par 2 la taille de la matrice stocke en mmoire. def get_n_most_similar_using_numpy(user_id, matrix,users, n_bests=20):
    """
    Get N sorted most similar profils for one user_id
    It use mergesort with worst complexity of nlog(n)
    @return 
        - index of similar profils
        - similarity score corresponding
    """
    index_user = users[users.user_id==user_id].index.values[0]
    sum_i_j = matrix[index_user][:] + matrix[:][index_user]
    idx_sorted = sum_i_j.argsort(kind="mergesort")[-n_bests:][::-1]
    return idx_sorted, sum_i_j[idx_sorted]

L'utilisation de l'algorithme mergesort est intressant pour des grands volumes de donnes car il est la plupart du temps plus efficace. Mais utilise plus de mmoire. Il faut choisir un algorithme respectant les contraintes de mmoire ou de rapidit.
